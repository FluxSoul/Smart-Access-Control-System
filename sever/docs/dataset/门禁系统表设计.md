根据你提供的详细字段需求，结合 PostgreSQL 的特性以及智能门禁系统的实际业务场景（如：考勤分析、高并发写入），我为你设计了以下的 SQL 建表语句。

这套设计重点解决了以下几个问题：
1.  **JSON 的使用**：利用 PG 强大的 JSONB 类型处理“允许通行的门列表”。
2.  **二进制存储**：使用 `BYTEA` 存储人脸和指纹特征值。
3.  **性能优化**：针对高频查询的字段（如用户ID、时间）建立了索引。

### 1. 设备表 (`devices`)
用于管理所有门禁终端，支持远程监控和固件升级。

```sql
CREATE TABLE devices (
    device_id VARCHAR(50) PRIMARY KEY,      -- 设备ID (如 MAC 地址或自定义编号)
    location VARCHAR(150) NOT NULL,         -- 安装位置 (如 "图书馆一楼东门")
    ip_address INET,                        -- IP地址 (使用 INET 类型支持 IP 查询)
    status VARCHAR(20) DEFAULT 'offline',   -- 在线状态 ('online', 'offline', 'maintenance')
    firmware_version VARCHAR(20),           -- 固件版本 (如 "v1.0.2")
    last_heartbeat TIMESTAMPTZ,             -- 最后心跳时间 (用于判断是否掉线)
    created_at TIMESTAMPTZ DEFAULT NOW()    -- 录入时间
);

-- 索引：加速按位置查找设备
CREATE INDEX idx_devices_location ON devices(location);
```

### 2. 人员权限表 (`user_permissions`)
这是核心表，存储身份特征和权限范围。

**设计亮点：**
*   **特征值存储**：使用 `BYTEA` 存储二进制特征数据（比 JSON 存 Base64 更节省空间且更安全）。
*   **权限列表**：使用 `JSONB` 存储允许通行的设备ID列表，方便快速查询和扩展。

```sql
CREATE TABLE user_permissions (
    user_id VARCHAR(50) PRIMARY KEY,        -- 用户ID (学号/工号)
    name VARCHAR(100) NOT NULL,             -- 姓名
    face_feature BYTEA,                     -- 人脸特征值 (二进制数据)
    fingerprint_feature BYTEA,              -- 指纹特征值 (二进制数据)
    
    -- 允许通行的门列表，使用 JSONB 格式，例如：["dev_001", "dev_002"]
    -- 如果为空数组或 NULL，表示无权限
    allowed_devices JSONB DEFAULT '[]'::jsonb, 
    
    valid_start TIMESTAMPTZ NOT NULL,       -- 有效期开始
    valid_end TIMESTAMPTZ NOT NULL,         -- 有效期结束
    is_active BOOLEAN DEFAULT TRUE,         -- 账号是否启用
    
    updated_at TIMESTAMPTZ DEFAULT NOW()    -- 更新时间
);

-- 索引：加速权限校验时的 JSON 查询 (查询某用户是否有权进入某设备)
CREATE INDEX idx_user_allowed_devices ON user_permissions USING GIN (allowed_devices);
```

### 3. 通行日志表 (`access_logs`)
数据量最大的表，设计时考虑了“写多读少”和“时间范围查询”的特性。

**设计亮点：**
*   **结果枚举**：明确区分成功/失败，便于统计安防数据。
*   **冗余设计**：虽然有关联 ID，但建议冗余 `user_name`（可选），防止用户删除后日志无法显示姓名。

```sql
CREATE TABLE access_logs (
    log_id BIGSERIAL PRIMARY KEY,           -- 日志ID (自增主键)
    user_id VARCHAR(50),                    -- 用户ID (可为空，如果是陌生人尝试开门则为空)
    device_id VARCHAR(50) NOT NULL,         -- 设备ID
    
    access_time TIMESTAMPTZ DEFAULT NOW(),  -- 通行时间
    auth_method VARCHAR(20) NOT NULL,       -- 通行方式: 'face', 'rfid', 'fingerprint', 'remote'
    result VARCHAR(10) NOT NULL,            -- 结果: 'success', 'failure'
    
    photo_url TEXT,                         -- 抓拍照片 URL (如果是人脸失败，存抓拍图)
    reason TEXT,                            -- 失败原因 (如 "权限不足", "未识别")
    
    created_at TIMESTAMPTZ DEFAULT NOW()    -- 入库时间
);

-- 核心索引：考勤和记录查询最常用的组合
CREATE INDEX idx_logs_user_time ON access_logs(user_id, access_time DESC);
-- 核心索引：设备端查询历史记录或管理员查某门的情况
CREATE INDEX idx_logs_device_time ON access_logs(device_id, access_time DESC);
-- 核心索引：按时间范围统计（如：今日通行人数）
CREATE INDEX idx_logs_time ON access_logs(access_time DESC);
```

### 💡 给你的额外建议

1.  **关于人脸/指纹特征值**：
    *   虽然你在需求中提到存 BLOB，但在 PostgreSQL 中对应的类型是 `BYTEA`。
    *   如果你的特征值是字符串形式的向量（例如 `[0.12, 0.99, ...]`），也可以考虑存为 `JSONB` 或者压缩后的文本，但 `BYTEA` 是最通用的二进制存储方式。

2.  **关于权限表的时间范围**：
    *   我在表中增加了 `valid_start` 和 `valid_end`。这在校园场景非常有用（例如：学生毕业自动失效，访客仅限当天有效），避免物理删除数据，只需更新状态即可。

3.  **日志表分区（进阶）**：
    *   如果这是一个长期运行的系统，日志表会迅速膨胀。建议在 PostgreSQL 中使用**声明式分区**，按**月**或**年**自动切分日志表。这样删除一年前的数据时，只需 `DROP` 对应的分区表，速度极快且不影响业务。
