在 Go 的分层架构（Handler -> Logic -> Repository）中，**Repository 层（Repo 层）** 的职责非常纯粹：**只负责与数据库打交道，执行 CRUD（增删改查）操作，不包含任何复杂的业务逻辑**。

基于你之前的数据库设计（设备表、人员权限表、通行日志表）以及智能门禁系统的业务场景，以下是这三张表在 Repo 层应该支持的具体操作接口设计：

### 1. 设备管理 Repo (`DeviceRepository`)
**职责**：管理硬件资产，支持设备注册、状态更新和基础信息查询。

```go
type DeviceRepository interface {
    // 1. 创建设备：新设备首次接入时注册
    Create(ctx context.Context, device *model.Device) error
    
    // 2. 更新设备信息：修改位置、IP 或固件版本
    Update(ctx context.Context, device *model.Device) error
    
    // 3. 获取设备详情：通过设备ID查询
    GetByID(ctx context.Context, deviceID string) (*model.Device, error)
    
    // 4. 更新心跳状态：设备上报心跳时调用，更新 last_heartbeat 和 status
    UpdateHeartbeat(ctx context.Context, deviceID string, status string) error
    
    // 5. 查询所有在线设备：用于后台监控大屏
    ListOnlineDevices(ctx context.Context) ([]*model.Device, error)
    
    // 6. 删除设备：设备报废或移除
    Delete(ctx context.Context, deviceID string) error
}
```
*   **关键点**：`UpdateHeartbeat` 是物联网系统特有的高频操作，用于判断设备是否掉线。

### 2. 人员权限 Repo (`UserPermissionRepository`)
**职责**：管理“人”和“权”，支持增删改查以及核心的权限校验。

```go
type UserPermissionRepository interface {
    // 1. 新增人员权限：录入新用户
    Create(ctx context.Context, user *model.UserPermission) error
    
    // 2. 更新人员信息：修改姓名、有效期或权限列表
    Update(ctx context.Context, user *model.UserPermission) error
    
    // 3. 获取人员详情：通过用户ID查询（含人脸/指纹特征值）
    GetByID(ctx context.Context, userID string) (*model.UserPermission, error)
    
    // 4. 删除人员权限：离职或毕业注销
    Delete(ctx context.Context, userID string) error
    
    // 5. 权限校验（核心）：检查某用户是否有某设备的通行权限
    // 逻辑：查询该用户，检查 allowed_devices JSONB 是否包含 targetDeviceID
    CheckPermission(ctx context.Context, userID string, targetDeviceID string) (bool, error)
    
    // 6. 查询过期或即将过期的用户：用于定时任务清理或提醒
    ListExpiringUsers(ctx context.Context, days int) ([]*model.UserPermission, error)
}
```
*   **关键点**：`CheckPermission` 是门禁系统的核心。虽然可以在 Logic 层查出数据再比对，但在 Repo 层利用 PostgreSQL 的 JSONB 操作符（如 `?` 或 `@>`）直接查询效率更高。

### 3. 通行日志 Repo (`AccessLogRepository`)
**职责**：负责海量数据的写入和统计分析。由于日志通常“只增不改”，所以没有 Update 操作。

```go
type AccessLogRepository interface {
    // 1. 新增日志：设备上报通行记录（最高频操作）
    Create(ctx context.Context, log *model.AccessLog) error
    
    // 2. 批量新增日志：如果设备支持离线缓存后批量上传
    CreateBatch(ctx context.Context, logs []*model.AccessLog) error
    
    // 3. 分页查询日志：后台管理界面展示历史记录
    // 支持按用户ID、设备ID、时间范围筛选
    List(ctx context.Context, query *model.LogQuery, page, pageSize int) ([]*model.AccessLog, int64, error)
    
    // 4. 统计通行次数：用于考勤或数据大屏
    // 例如：统计某用户今日进入次数
    CountByCondition(ctx context.Context, query *model.LogQuery) (int64, error)
    
    // 5. 清理旧数据：定时任务调用，删除 N 天前的日志
    DeleteBefore(ctx context.Context, time time.Time) error
}
```
*   **关键点**：
    *   `Create` 必须极其高效。
    *   `List` 需要构建动态 SQL（使用 `sqlx` 或 `gorm` 的条件构建），因为查询条件组合很多（查某人的、查某门的、查失败记录的等）。

### 💡 架构示例：如何在 Logic 层组合使用

为了让你更清楚 Repo 层如何服务于业务，这里有一个**“设备上报通行记录”**的 Logic 层伪代码示例：

```go
// Logic 层：处理开门记录上报
func (l *AccessLogic) HandleAccessReport(req *dto.AccessReportReq) error {
    // 1. 调用 Repo 层：记录日志（无论成功失败都要记）
    logModel := &model.AccessLog{
        UserID: req.UserID,
        DeviceID: req.DeviceID,
        Result: req.Result, // success / failure
        // ...
    }
    if err := l.logRepo.Create(l.ctx, logModel); err != nil {
        return err
    }

    // 2. 业务逻辑：如果是开门成功，可能需要更新设备状态或其他联动
    if req.Result == "success" {
        // 比如：更新 Redis 中的“最后通行时间”供前端实时推送
    }
    
    return nil
}
```

### 总结
在 Repo 层，你的关注点应该是 **SQL 的 CRUD** 和 **数据结构的映射**，不要在这里写 `if/else` 业务判断。

*   **DeviceRepo**：侧重设备状态的维护。
*   **UserRepo**：侧重特征的存取和权限的匹配。
*   **LogRepo**：侧重高吞吐的写入和灵活的查询。
