# 重庆邮电大学2023级物联网工程专业-物联网复杂工程课程设计需求分析

*仅为前期可行性分析，不代表最终产品*

题目：智能出入管理系统

## 要求文档分析

### 应用场景

应用于校园、小区和房屋的安全管理与信息采集。是解决重要部门出入口实现安全防范管理的有效措施，有利于提高社会治安治理水平。

设计智能出入管理系统，实现门禁控制，数据库存储相关信息与基于数据分析的警报提醒功能。

三大部分：
1. 边缘数据采集：包括RFID身份认证、人脸识别、智慧门禁等内容。
2. 数据网络传输：借助WiFi网络等，实现边缘层采集数据的上报和后台分析数据发送等。
3. 后台数据分析：基于远端服务器或者网路云平台等方式建立后台数据库，实现存储和分析边缘层所采集的数据。

### 核心功能要求与指标

1. 具备基于人脸识别的门禁管理功能，可以有效监控出入人员身份信息，识别准确率不低于85%。
2. 具备 RFID 读取，以及密码键盘、指纹识别、手机APP远程开关门禁等中的至少一种功能，可以有效监控出入人员身份信息。
3. 具备闸机开关、语音播报等功能，能对人员进出的各种情况进行相应和提示。
4. 开发门禁智能看板，显示日期，天气以及区域通知公告等内容。
5. 基于WiFi或者4G、5G等搭建数据传输网络，实现边缘节点间以及与数据管理中心间等交互，数据丢包率不高于1%，响应时间应控制在3S之内。
6. 基于云平台或者远程服务器搭建数据库，对人员进出权限进行集中管理，同时对人员出入方式进行分类统计。
7. 系统具备报警推送功能，一旦数据管理系统发现异常情况，立即启动报警信息报送，以便采取紧急应对措施，误报率不高于1%。

## 实际需求分析

### 1. 产品描述

- 产品愿景：一个基于AI的现代化智能门禁系统。
- 目标用户：有出入口管理需求的单位。
- 运行环境：城市户外环境。

### 2. 硬件需求

#### 流程描述

- 流程描述：
	1. 门禁处于休眠状态，但维护红外传感器与MQTT远程访问心跳。
	2. 门禁红外传感器模块检测到访问者靠近，进入工作状态，打开以下模块：
		1. 摄像头模块；
		2. 显示屏模块；
		3. 指纹识别模块；
		4. 语音播报模块；
	3. 访问者使用摄像头进行人脸识别访问。
		1. 或者使用指纹识别模块；
	4. 进行识别，成功，打开门禁，向远程后台发送访问消息。
		- 如果出现告警条件，锁定系统，语音播报警告，向后台发送报警信息，通知管理者。
	5. 重新进入休眠。

#### 硬件选型

核心需求：边缘计算，人脸识别。

采用“双核架构”，即一块 ESP32S3 和 STM32F103C8T6，具体负责模块如下。

##### 其他注意事项

1. **电平匹配**：
    - ESP32-S3 和 STM32F103 都是 3.3V 逻辑。
    - 但是，**MFRC522** 和 **SYN6288** 有些模块是 5V 供电的。如果它们是 5V 供电，它们的 TX 引脚输出可能是 5V 电平，**直接连到 STM32 的 RX 可能会烧毁引脚**。
    - **解决**：购买模块时确认是 3.3V 版本，或者在 RX/TX 线上串联一个 1kΩ 电阻进行限流保护。

2. **通信协议**：
    - ESP32-S3 与 STM32 之间通过 **UART** 通信。
    - 建议波特率设为 **115200**。
    - 定义简单的指令集，例如：
        - STM32 发给 ESP32：`CARD:OK` (刷卡成功), `FP:OK` (指纹成功)
        - ESP32 发给 STM32：`OPEN:1` (开锁), `SAY:Hello` (播报语音)

### 3. 软件需求

#### 硬件软件基本框架

| 角色          | 硬件载体      | 核心职责                                                                                      | 运行环境               |
| :---------- | :-------- | :---------------------------------------------------------------------------------------- | :----------------- |
| 大脑 (Master) | ESP32-S3  | 1. 运行 LVGL 图形界面  <br>2. 跑人脸识别 AI 模型  <br>3. 处理 WiFi/MQTT 联网通信  <br>4. 业务逻辑决策<br>5. 控制语音播报 | FreeRTOS (ESP-IDF) |
| 管家 (Slave)  | STM32F103 | 1. 实时轮询传感器 (指纹/RFID)  <br>2. 驱动继电器开锁  <br>3. 电源管理/看门狗                                     | 裸机 (While Loop)    |

**物理连接**：ESP32 TX2 <-> STM32 RX2 | ESP32 RX2 <-> STM32 TX2  
**波特率**：115200

##### 协议格式定义

建议采用：`帧头 + 命令字 + 数据 + 校验/帧尾` 的简化版，或者直接用 **JSON** (如果内存允许) 或 **定长字符串**。  
为了简单高效，推荐使用 **冒号分隔的字符串**：

`CMD:DATA\r\n`

#### 指令集清单

**A. STM32 -> ESP32 (上报状态)**

- `FP:SUCCESS`：指纹验证通过
- `RFID:OK`：刷卡成功
- `PIR:1`：检测到有人靠近（唤醒信号）
- `PIR:0`：人离开
- `SYS:READY`：STM32 启动完成
- `VOICE:WELCOME`：播放“欢迎光临”
- `VOICE:ACCESS_DENIED`：播放“验证失败”

**B. ESP32 -> STM32 (下发指令)**

- `LOCK:OPEN`：执行开锁动作（吸合继电器 3 秒）
- `LED:RGB(255,0,0)`：控制指示灯颜色（如果有）

##### 1. STM32 端软件设计 (管家)

STM32 的代码逻辑相对简单，追求**实时性**。推荐使用 **状态机** 模式。

- **初始化**：配置串口、GPIO、定时器。
- **主循环**：
    1. **传感器轮询**：
        - 检查 UART 是否有指纹模块数据。
        - 检查 SPI 是否有 RFID 数据。
        - 读取 HC-SR501 电平。
    2. **串口接收处理**：
        - 使用**中断接收 + 环形缓冲区**处理 ESP32 发来的指令。
        - 解析指令（如收到 `LOCK:OPEN` -> 拉高继电器引脚 -> 延时 -> 拉低）。
    3. **执行动作**：
        - 一旦检测到验证成功（指纹/刷卡），立即通过串口向 ESP32 发送 `FP:SUCCESS`。

##### 2. ESP32-S3 端软件设计 (大脑)

ESP32 的逻辑复杂，必须使用 **FreeRTOS** 进行多任务调度。

- **任务 1：UI 显示任务**
    - 运行 **LVGL** 图形库。
    - 负责显示时间、天气、看板公告。
    - 响应触摸事件。
- **任务 2：AI 视觉任务**
    - 监听摄像头数据流。
    - **策略**：平时不运行 AI 以省电。当收到 STM32 发来的 `PIR:1`（有人）信号时，唤醒摄像头进行人脸检测。
    - 调用 ESP-WHO 库进行人脸比对。
- **任务 3：通信与联网任务**
    - **下行**：监听 STM32 的串口数据。收到 `FP:SUCCESS` 后，触发开锁逻辑。
    - **上行**：连接 WiFi，通过 MQTT 协议向服务器发送日志（如“张三 12:00 进门”）。
    - 接收服务器下发的远程开锁指令。

##### 核心业务流程图解

- 场景一：刷脸开门

1. **感知**：HC-SR501 检测到人 -> STM32 发送 `PIR:1`。
2. **唤醒**：ESP32 收到信号 -> 点亮屏幕 -> 开启摄像头。
3. **识别**：ESP32 抓拍 -> AI 比对 -> 确认是“管理员”。
4. **决策**：ESP32 发送 `LOCK:OPEN` 给 STM32 + `VOICE:WELCOME` 。
5. **执行**：STM32 驱动继电器开锁，驱动语音模块播报。
6. **上报**：ESP32 通过 WiFi 上传记录到服务器。

- 场景二：指纹开门 (离线可用)

1. **感知**：STM32 轮询发现指纹模块有数据。
2. **验证**：STM32 内部比对指纹特征值（指纹模块自带算法）。
3. **上报**：STM32 发送 `FP:SUCCESS`  + `VOICE:WELCOME`给 ESP32。
4. **决策**：ESP32 收到成功信号 -> 发送 `LOCK:OPEN` 回 STM32。
5. **执行**：STM32 开锁。
6. **反馈**：ESP32 在屏幕上显示“验证成功”动画。

##### 其他注意事项

1. **看门狗**：在 STM32 端务必开启**独立看门狗**。如果 STM32 死机，ESP32 将无法控制门锁，系统会彻底瘫痪。
2. **电源管理**：在软件中加入**低功耗逻辑**。
    - 当 `PIR:0` 持续 30 秒 -> ESP32 关闭屏幕背光，降低 CPU 频率，进入 Light Sleep。
    - STM32 保持低功耗运行，仅监听 PIR 和 串口唤醒中断。
3. **数据同步**：如果 ESP32 断网了，STM32 依然能开门，但无法上传记录。STM32 可以在本地 Flash (EEPROM) 简单存储最近 50 条记录，等 ESP32 联网后，通过串口请求补传数据。

### 3. 服务器端需求

设计参考[Modular and Highly Scalable IoT Node System: End-to-End Architecture Design from Edge Sensing to Cloud Applications](https://foreveryang.cn/blog/mokuaihuagaokuozhanxingwulianwangjiedianxitongcongbianyuanganzhidaoyunduanyingyongdeduandaoduanjiagousheji-0)

需要增加的表
- **设备表**：存储门禁的基本信息。
    - 字段：`设备ID`、`安装位置`、`IP地址`、`在线状态`、`固件版本`。
- **人员权限表**：存储谁能进哪扇门。
    - 字段：`用户ID`、`姓名`、`人脸特征值(BLOB)`、`指纹特征值`、`允许通行的门列表(JSON)`、`有效期`。
- **通行日志表**：数据量最大，用于分析和考勤。
    - 字段：`日志ID`、`用户ID`、`设备ID`、`通行时间`、`通行方式`（人脸/卡/指纹）、`抓拍照片URL`、`结果`（成功/失败）。

智能看板数据流（WebSocket）
- **日期/时间**：服务器通过 NTP 协议同步给设备，或者设备启动时请求服务器时间。
- **天气**：服务器调用第三方天气 API（如和风天气），每天定时获取，缓存到数据库。设备通过 MQTT 或 HTTP 轮询获取。
- **通知公告**：管理员在 Web 端发布公告 -> 写入数据库 -> 服务器通过 MQTT 推送 `NOTICE_UPDATE` 消息给设备 -> 设备刷新屏幕。
